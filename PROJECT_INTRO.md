# DOORS 科技规划追溯助手 — 项目介绍

## 一句话概述

**"用聊天的方式，追溯科技规划中每一项需求、每一个任务、每一个项目的来龙去脉。"**

DOORS 科技规划追溯助手是一个基于知识图谱 + 大语言模型（LLM）的智能问答与可视化系统。用户只需用自然语言提问，系统就能自动检索数据库中的科技规划信息，并以**交互式知识图谱**的形式直观展示需求、大纲、文本、任务、科研项目之间的完整追溯链路。

---

## 我们解决了什么问题？

在大型科技规划管理中，一个核心痛点是：**信息是分散且割裂的**。

- 一个"科技规划需求"对应着多个"大纲"，每个大纲下又有多个"文本"，文本规定了具体"任务"，任务最终由"科研项目"落地执行
- 管理者想知道"某个需求最终落实到了哪些项目？"或者"某个项目是为了满足哪个需求？"，传统方式需要人工在多个系统、多张 Excel 表中逐一查找、手动拼接
- 这个过程耗时耗力，还容易遗漏关键关联

我们的系统通过**知识图谱**将这些分散的数据串联成一张"追溯网"，再结合**大语言模型**让用户用自然语言直接提问，几秒钟内完成过去需要几小时的人工追溯工作。

## 技术路线

### 整体架构

```
用户浏览器（React 前端）
    │  自然语言提问
    ↓
Dify AI 编排平台（Agent 模式）
    │  LLM 理解意图 → 生成 Cypher 图查询语句
    ↓
Neo4j 代理（FastAPI 安全网关）
    │  安全校验 → 转发查询
    ↓
Neo4j 图数据库（云端 Aura 实例）
    │  返回结构化图数据
    ↓
前端知识图谱可视化（Cytoscape.js）
    │  交互式节点-关系图谱
    ↓
用户直观看到完整追溯链路
```

### 核心技术组件

| 组件 | 技术选型 | 职责 |
|------|---------|------|
| **前端** | React 19 + TypeScript + Vite + TailwindCSS 4 | 对话界面 + 知识图谱渲染 |
| **图谱可视化** | Cytoscape.js | 交互式节点-关系图，支持缩放、拖拽、多布局切换 |
| **AI 编排** | Dify（Agent 模式） | 管理 LLM 调用 + 工具调度 + 对话上下文 |
| **大语言模型** | Gemini 2.0 Flash | 理解用户意图，自动生成图数据库查询语句 |
| **安全代理** | FastAPI（Python） | 查询安全校验（禁止写操作）+ 数据格式转换 |
| **图数据库** | Neo4j Aura（云端） | 存储科技规划实体及其追溯关系 |

### 数据模型 — 追溯链路的"骨架"

我们将科技规划中的 5 类核心实体建模为图节点，用 6 种关系将它们串联：

```
科技规划需求 ──关联──→ 科技规划大纲 ──包含──→ 科技规划文本 ──规定──→ 科技规划任务 ──实现──→ 科研项目
    │                                                                              │
    └──────────── 需求细化（子需求）─────────────────────────────────────────────────┘
                                                                                   │
                                          任务依赖 ←─────────────────────────────────┘
```

这种图结构天然支持**双向追溯**：
- **向下追溯**：从某个需求出发，一路追踪到最终落地的科研项目
- **向上追溯**：从某个项目反查，回溯到它满足的原始需求

### 工作流程

1. **用户提问**：在聊天框输入自然语言问题，如"查询所有高优先级需求，并追溯它们最终对应哪些科研项目"
2. **AI 理解与规划**：LLM（Gemini）分析问题语义，自动规划需要查询哪些节点和关系
3. **生成图查询语句**：LLM 生成 Neo4j 专用的 Cypher 查询语句
4. **安全执行**：查询经安全代理校验（确保只读），转发到 Neo4j 执行
5. **图谱可视化**：查询结果以交互式知识图谱展示，节点按类型着色，关系用箭头连线
6. **持续对话**：支持多轮追问，保持对话上下文

## 相比传统 RAG 模式的核心优势

> **RAG（检索增强生成）** 是当前大模型应用的主流方案：将文档切片存入向量数据库，用户提问时检索相关片段，拼接后交给 LLM 生成回答。

我们的方案选择了 **Graph + LLM** 而非 RAG，原因如下：

### 1. 从"碎片化检索"到"结构化追溯"

| 对比维度 | 传统 RAG | DOORS 图追溯方案 |
|---------|---------|-----------------|
| 数据组织 | 文档被切成碎片，上下文丢失 | 实体和关系完整保留在图结构中 |
| 回答方式 | 从碎片中拼凑答案，可能遗漏关键关联 | 沿关系链精准追溯，一步不落 |
| 典型短板 | 无法回答"A 最终关联到哪些 C？" | 天然支持多跳关系查询 |

**打个比方**：RAG 像是把一本书撕成纸条，然后从中挑出相关纸条拼答案；我们的方案则是保留了书的目录结构和章节关系，能准确告诉你"第3章第2节和第5章的关系是什么"。

### 2. 从"猜相关"到"确相关"

- **RAG** 依赖向量相似度来判断"哪些内容相关"——但语义相似 ≠ 逻辑相关。比如"发动机故障"和"发动机维修"语义相似度高，但可能分属不同项目，没有直接的追溯关系
- **图方案** 中的关系是显式存储的——"需求 A → 大纲 B → 任务 C"这条链是确定的，不存在"猜"的成分

### 3. 可视化带来可解释性

- **RAG** 的输出是一段文字回答，用户看不到信息之间的关联
- **图方案** 将追溯结果可视化为知识图谱，用户可以**一眼看清**：
  - 哪些节点参与了这个追溯链？
  - 它们之间是什么关系？
  - 是否有遗漏的中间环节？
  - 某条链路是否完整？

### 4. 精确查询，消除幻觉

- **RAG** 存在"检索盲区"——如果相关文档片段没有被向量检索命中，LLM 可能会编造信息（幻觉）
- **图方案** 中 LLM 生成的是**结构化查询语句**（Cypher），数据库返回的是**精确的结构化数据**，LLM 只负责"翻译成人话"，大大降低了幻觉风险
- 用户可以点击"查看 Cypher 语句"和"查看思考过程"，完全透明地了解 AI 是如何得出这个结果的

### 5. 真正支持"追溯"场景

"追溯"的核心需求是**沿着关系链一步步走**：
- "这个需求→对应哪些大纲？→每个大纲包含哪些文本？→每个文本规定了哪些任务？→每个任务由哪个项目执行？"
- 这是典型的多跳图遍历，RAG 方案需要多次检索+拼接，效率低且容易断链
- 图方案一条 Cypher 语句即可完成，精确且高效

### 6. 数据更新友好

- **RAG**：文档更新后需要重新切片、重新向量化、重新入库，维护成本高
- **图方案**：数据库中新增/修改一个节点或关系，查询结果即时反映变化，无需额外处理

## 应用场景

- **科技规划管理者**：快速了解某项规划需求的落地情况，识别"悬空需求"（没有对应项目的需求）
- **科研项目负责人**：追溯项目立项依据，确认项目满足的原始需求
- **审计与合规**：验证科技规划从需求到执行的全链路完整性
- **决策支持**：通过图谱直观发现需求密度、任务依赖关系等宏观模式

## 部署方式

系统采用前后端分离 + 云服务架构，部署轻量、维护简单：

| 组件 | 部署平台 | 说明 |
|------|---------|------|
| 前端 | GitHub Pages | 免费静态托管，推送即部署 |
| AI 编排 | Dify Cloud | 托管的 Agent 平台，无需自建 |
| 安全代理 | Render | Python 微服务，支持免费额度 |
| 图数据库 | Neo4j Aura | 云端图数据库，免运维 |

## 总结

DOORS 科技规划追溯助手不是简单的"文档问答"工具，而是一个**以知识图谱为核心、以大模型为交互入口的追溯分析系统**。它将原本割裂在多个系统中的科技规划数据，组织成一张可视化的"关系网"，让管理者能够像聊天一样自然地探索数据之间的深层联系，真正实现了**"问得自然、查得精准、看得明白"**。
